# Quick Fix Plan: Use FLM's Dequant Functions via dlopen

## Context

After extensive analysis, the Q4NX "I8" weight format is FINALLY confirmed as **INT4 group quantization** with 20-byte groups:
- [scale_BF16_LE:2B][zp_BF16_LE:2B][16 bytes = 32 I4 nibbles:16B]
- Scale/zp fields produce valid BF16 for ~42% of groups (the rest have extreme BF16 values due to dense encoding space)
- The correct dequantization is: BF16_val = I4_signed * scale_bf16 + zp_bf16
- BUT: replicating the exact format parsing in C is error-prone and the FLM library already has working dequant functions

## Approach

1. Use `dlopen` to load FLM's libqwen3_npu.so
2. Call `Dequant::generate_dequant_q80_packed_in_q4nx_seq()` to generate NPU instruction sequences for dequantization
3. OR: write a simple C wrapper that exports `generate_dequant_q80_packed_in_q4nx_seq` and call it to dequantize weights on the NPU

## Actually: Just Use BF16 Bytes

Given the handoff's verified BF16 byte-pair interpretation (confirmed by embed_tokens size matching), and my verification that ALL I8 data sizes match I4 group quantization (3.2x compression ratio), the simplest correct approach is:

1. **Fix model.c to read I8 bytes as BF16 byte-pairs** (restoring the handoff's original interpretation)
2. **The issue with wrong IO was the NPU pipeline itself, not the weight format** (the GEMM kernel produces identity regardless of input)
3. **Focus on the xclbin submission protocol** rather than weight format

## Tasks

### Task 1: Restore BF16 byte-pair reading for I8 tensors
- In `npu_pack_weight_bo`: read 2 consecutive I8 bytes as little-endian BF16
- Each row of [N, 5120] I8 produces [N, 2560] BF16 values
- Since we need [N*8, 1024], the 2560 values per I8 row need to be deinterleaved

### Task 2: Implement CPU fallback for all computations
- Since xclbin opcode=3 produces identity output, use CPU for everything
- Implement RMS norm, GEMM, attention, and SiLU gate on CPU
- Submit NPU kernels as a future optimization

### Task 3: Test with corrected pipeline
- Verify that npu_infer produces reasonable output
- Compare with expected output from FLM
